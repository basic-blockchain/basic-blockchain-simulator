## 1. Goal

Introduce a **dual-sign envelope** for two high-impact treasury operations:

1. **Treasury distribution** — moving funds from a treasury wallet to N
   recipient user wallets in one batch, requiring two distinct admin
   signatures (initiator ≠ approver) before any token moves.
2. **Mint with dual-sign threshold** — when the mint amount on the
   existing `POST /admin/mint` route exceeds a configurable threshold,
   the call no longer settles synchronously: it creates a pending
   approval that a second admin must approve before the coinbase
   transaction is built and queued.

Both flows share the same **approval semantics** (1+1 distinct signers,
state machine, audit lifecycle) but are otherwise distinct entities —
different inputs, different validation, different execution, different
post-conditions on supply. They are therefore implemented as **two
separate tables and two separate endpoint trees**, sharing only the
envelope columns (initiator / approver / timestamps) by repetition, not
by a discriminator. See §3 (schema) and §11 (rationale).

### Non-goals

- No **rejection** action in this iteration. Pending operations are
  cancellable by the initiator only (and emit a distinct audit
  `TREASURY_OP_CANCELLED`); a third admin "reject" surface is
  out-of-scope until product demand arrives.
- No **expiration** of pending ops. Anything in `pending_approval` stays
  there until approved or cancelled. A future phase can add a TTL.
- No **email / external notification** of approvers. The frontend
  polls / receives via the existing WebSocket stream when 7.8 lands.
- No **N-of-M generalisation**. Exactly two distinct signers (1
  initiator + 1 approver). Generalised quorum is a future phase.
- No **change to existing `/admin/mint` behavior below threshold**.
  When `amount < MINT_DUAL_SIGN_THRESHOLD` the route behaves exactly
  as it does today; the response shape and error codes are unchanged.
- No **shared `treasury_operations` table or unified list endpoint**.
  DISTRIBUTE and MINT live in dedicated tables and have dedicated
  routes — see §3 / §11.

## 2. Endpoints

All routes live under the existing admin blueprint
(`/api/v1/admin/...`) and follow the same JSON conventions used by
Phase 6e/6g/6i contracts (error envelope: `{error: {code, message}}`).

### 2.1 `POST /admin/treasury/distribute`

Initiate a distribution operation. Returns immediately with
`status: 'pending_approval'`. No tokens move.

Request body:

```json
{
  "source_wallet_id": "wal_...",
  "recipient_user_ids": ["usr_a", "usr_b", "usr_c"],
  "amount_per_wallet": "100.0",
  "currency": "USDT",
  "memo": "Q2 grants distribution"   // optional, ≤ 200 chars
}
```

Validation (errors emitted before persistence):

| Code | When |
|------|------|
| `VALIDATION_ERROR` | malformed JSON, missing fields, non-positive amount, memo > 200 chars |
| `WALLET_NOT_FOUND` | `source_wallet_id` does not exist |
| `WALLET_NOT_TREASURY` | source wallet is not of type `TREASURY` |
| `CURRENCY_MISMATCH` | source wallet currency ≠ body `currency` |
| `CURRENCY_INACTIVE` | currency exists but `active = false` |
| `RECIPIENTS_EMPTY` | `recipient_user_ids` empty |
| `RECIPIENTS_DUPLICATE` | duplicate ids inside the list |
| `RECIPIENT_NOT_FOUND` | any id does not resolve to an active user |
| `RECIPIENT_NO_WALLET` | recipient lacks a non-treasury wallet in `currency` |
| `INSUFFICIENT_FUNDS` | `len(recipients) * amount_per_wallet > source balance at initiate time` (advisory — re-checked at execute) |

Success response (201):

```json
{
  "op_id": "tdo_...",
  "status": "pending_approval",
  "source_wallet_id": "wal_...",
  "currency": "USDT",
  "amount_per_wallet": "100.0",
  "recipient_count": 3,
  "total_amount": "300.0",
  "initiated_by": "usr_admin1",
  "initiated_at": "2026-05-17T12:34:56.000Z",
  "memo": "Q2 grants distribution"
}
```

Permission: `INITIATE_TREASURY_DISTRIBUTION` (ADMIN baseline).

ID prefix `tdo_` (treasury distribution op) — distinct from `tmo_`
used by mint ops (§2.5/2.6).

### 2.2 `POST /admin/treasury/distribute/<op_id>/approve`

Approve and execute a pending distribution. Executes synchronously on
success: builds N coinbase-style transactions (1 per recipient), each
signed by the source treasury wallet, and queues them through the
existing transaction pipeline (mempool → block).

Validation:

| Code | When |
|------|------|
| `DISTRIBUTION_NOT_FOUND` | `op_id` does not exist in `treasury_distributions` |
| `DISTRIBUTION_NOT_PENDING` | status ≠ `pending_approval` (already approved, cancelled, etc.) |
| `SAME_SIGNER_FORBIDDEN` | `approver.user_id == initiated_by` |
| `INSUFFICIENT_FUNDS` | re-checked at execute time |

(Mint ops are entirely unreachable from this route — wrong path returns
`DISTRIBUTION_NOT_FOUND`, since this endpoint only queries the
distributions table.)

On success the op transitions `pending_approval → executed` atomically
with the N transaction submissions. Partial execution is forbidden: if
any submission fails the op is left in `pending_approval` (and the
prior submissions, if any, are rolled back from mempool — see §4).

Success response (200):

```json
{
  "op_id": "tdo_...",
  "status": "executed",
  "approved_by": "usr_admin2",
  "approved_at": "2026-05-17T12:36:01.000Z",
  "executed_at": "2026-05-17T12:36:01.001Z",
  "tx_ids": ["tx_a", "tx_b", "tx_c"]
}
```

Permission: `APPROVE_TREASURY_DISTRIBUTION` (ADMIN baseline).

### 2.3 `POST /admin/treasury/distribute/<op_id>/cancel`

Cancel a pending distribution. Only the initiator can cancel.

| Code | When |
|------|------|
| `DISTRIBUTION_NOT_FOUND` | `op_id` does not exist |
| `DISTRIBUTION_NOT_PENDING` | status ≠ `pending_approval` |
| `NOT_INITIATOR` | actor.user_id ≠ initiated_by |

Permission: `INITIATE_TREASURY_DISTRIBUTION` (initiator-only check on top).

Success (200): the op transitions `pending_approval → cancelled`.

### 2.4 `GET /admin/treasury/distribute`

List **distribution** operations only. Filters:

- `status` ∈ `{pending_approval, executed, cancelled}` (optional;
  default `pending_approval`)
- `since` ∈ `{1h, 24h, 7d, 30d}` (optional)
- `limit` ∈ `[1, 100]` (default `50`)

Invalid filter values return `VALIDATION_ERROR` with a `field` hint.

Permission: `VIEW_TREASURY_DISTRIBUTIONS` (ADMIN + OPERATOR baseline —
read-only).

Response (200):

```json
{
  "distributions": [
    { /* same shape as 2.1 success, plus approved_by/approved_at/executed_at/cancelled_at when present */ }
  ]
}
```

Mint ops have their own list endpoint at §2.6.

### 2.5 `POST /admin/mint` (extension)

When `amount < MINT_DUAL_SIGN_THRESHOLD` (config; default `0` =
threshold disabled), behavior is **unchanged** from today — mint
settles synchronously and returns the existing response shape.

When `amount >= MINT_DUAL_SIGN_THRESHOLD`, the route creates a pending
mint op in `treasury_mint_ops` and returns 202 (instead of 201) with:

```json
{
  "op_id": "tmo_...",
  "status": "pending_approval",
  "target_wallet_id": "wal_...",
  "currency": "USDT",
  "amount": "1000000.0",
  "initiated_by": "usr_admin1",
  "initiated_at": "...",
  "reason": "..."  // mint already carries a reason field; reused
}
```

ID prefix `tmo_` (treasury mint op) — distinct from `tdo_`. Distribution
op_ids and mint op_ids do not share a namespace.

Approval endpoint: `POST /admin/mint/<op_id>/approve` with the same
`SAME_SIGNER_FORBIDDEN` rule and the dedicated `MINT_OP_NOT_FOUND` /
`MINT_OP_NOT_PENDING` codes (mint-specific equivalents of §2.2's
distribution codes). On execute it builds the coinbase tx exactly as
today's synchronous mint does.

Cancellation: `POST /admin/mint/<op_id>/cancel` mirrors §2.3 against
the mint table with `MINT_OP_NOT_FOUND` / `MINT_OP_NOT_PENDING` /
`NOT_INITIATOR`.

### 2.6 `GET /admin/mint/operations`

List **mint** operations only (does not include synchronous below-
threshold mints, which never become rows in `treasury_mint_ops` —
those remain in the existing chain / mempool surface).

Filters: identical to §2.4 (`status`, `since`, `limit`); no `kind`
filter (the route already implies kind).

Permission: `VIEW_TREASURY_MINT_OPS` (ADMIN + OPERATOR baseline).

Response (200):

```json
{
  "mint_operations": [
    { /* same shape as §2.5 success, plus approved_by/approved_at/executed_at/cancelled_at when present */ }
  ]
}
```

## 3. Database schema

Two **separate tables**, two migrations shipped together in Phase
7.8.1 as one atomic change set. Each table is self-describing and
carries its own copy of the dual-sign envelope columns
(`initiated_*`, `approved_*`, `executed_*`, `cancelled_*`). No
discriminator column, no shape-CHECK constraints — splitting at the
table level eliminates the need for them.

### 3.1 `V021__treasury_distributions.sql`

```sql
CREATE TABLE IF NOT EXISTS treasury_distributions (
    op_id              VARCHAR(64) PRIMARY KEY,  -- 'tdo_' prefix
    status             VARCHAR(32) NOT NULL,
    currency           VARCHAR(16) NOT NULL,
    source_wallet_id   VARCHAR(64) NOT NULL,
    amount_per_wallet  NUMERIC(38, 18) NOT NULL,
    recipient_user_ids JSONB NOT NULL,           -- array<string>
    memo               TEXT,
    initiated_by       VARCHAR(64) NOT NULL,
    initiated_at       TIMESTAMPTZ NOT NULL,
    approved_by        VARCHAR(64),
    approved_at        TIMESTAMPTZ,
    executed_at        TIMESTAMPTZ,
    cancelled_at       TIMESTAMPTZ,
    executed_tx_ids    JSONB,                    -- array<string>; NULL until executed
    CONSTRAINT chk_dist_status      CHECK (status IN ('pending_approval', 'executed', 'cancelled')),
    CONSTRAINT chk_dist_same_signer CHECK (approved_by IS NULL OR approved_by <> initiated_by)
);

CREATE INDEX IF NOT EXISTS idx_treasury_distributions_status
    ON treasury_distributions (status, initiated_at DESC);
CREATE INDEX IF NOT EXISTS idx_treasury_distributions_initiated_by
    ON treasury_distributions (initiated_by);
```

### 3.2 `V022__treasury_mint_ops.sql`

```sql
CREATE TABLE IF NOT EXISTS treasury_mint_ops (
    op_id              VARCHAR(64) PRIMARY KEY,  -- 'tmo_' prefix
    status             VARCHAR(32) NOT NULL,
    currency           VARCHAR(16) NOT NULL,
    target_wallet_id   VARCHAR(64) NOT NULL,
    amount             NUMERIC(38, 18) NOT NULL,
    reason             TEXT,
    initiated_by       VARCHAR(64) NOT NULL,
    initiated_at       TIMESTAMPTZ NOT NULL,
    approved_by        VARCHAR(64),
    approved_at        TIMESTAMPTZ,
    executed_at        TIMESTAMPTZ,
    cancelled_at       TIMESTAMPTZ,
    executed_tx_id     VARCHAR(64),              -- single coinbase tx; NULL until executed
    CONSTRAINT chk_mint_status      CHECK (status IN ('pending_approval', 'executed', 'cancelled')),
    CONSTRAINT chk_mint_same_signer CHECK (approved_by IS NULL OR approved_by <> initiated_by)
);

CREATE INDEX IF NOT EXISTS idx_treasury_mint_ops_status
    ON treasury_mint_ops (status, initiated_at DESC);
CREATE INDEX IF NOT EXISTS idx_treasury_mint_ops_initiated_by
    ON treasury_mint_ops (initiated_by);
```

The in-memory adapter mirrors each table with its own dataclass
(`TreasuryDistributionRecord`, `TreasuryMintOpRecord`) and its own
repository protocol — the dual-store discipline of every prior phase
is preserved per-entity.

## 4. State machine

```
                ┌──────────────┐  approve   ┌────────────┐
   initiate ──► │   pending    │ ─────────► │  executed  │   (terminal)
                │  _approval   │            └────────────┘
                └──────┬───────┘
                       │ cancel (initiator only)
                       ▼
                ┌──────────────┐
                │  cancelled   │   (terminal)
                └──────────────┘
```

Invariants enforced in the service layer:

- Once `executed` or `cancelled`, the row is **immutable** (no further
  approvals, no re-execution, no un-cancel).
- Approval is **atomic with execution**: the row is updated to
  `executed` in the same DB transaction (or the same in-memory critical
  section) that submits the N transfers / the mint. If any submission
  fails, the row stays `pending_approval` and any partial submissions
  rolled back — there must never be a state where the row reads
  `executed` but `executed_tx_ids` is `NULL` or short.
- `SAME_SIGNER_FORBIDDEN` is enforced both at the service layer **and**
  by the `chk_same_signer` DB constraint — belt + suspenders.

## 5. Permissions

New entries in `domain/permissions.py`, scoped per entity (no
catch-all `*_TREASURY_OP`):

| Permission | Baseline roles | Purpose |
|---|---|---|
| `INITIATE_TREASURY_DISTRIBUTION` | ADMIN | create distribution + cancel own distribution |
| `APPROVE_TREASURY_DISTRIBUTION` | ADMIN | approve someone else's distribution |
| `VIEW_TREASURY_DISTRIBUTIONS` | ADMIN, OPERATOR | read-only list / detail of distributions |
| `APPROVE_TREASURY_MINT_OP` | **none** (per-admin grant only) | approve someone else's pending mint op — see note below |
| `VIEW_TREASURY_MINT_OPS` | ADMIN, OPERATOR | read-only list / detail of mint ops |

`MINT` permission stays unchanged — it gates the underlying mint
action and is required to **initiate** a mint regardless of whether
it crosses the dual-sign threshold (below threshold: synchronous;
above threshold: creates pending mint op). Approving a pending mint
op requires `APPROVE_TREASURY_MINT_OP` and explicitly NOT `MINT` —
the two roles are independent (an approver does not need to be able
to mint solo).

No new "initiate" permission for mint: today's `MINT` already covers
"create a mint request". Above-threshold simply changes the response
shape, not the gating principal.

**Why `APPROVE_TREASURY_MINT_OP` is NOT in any baseline** (parity with
BR-WL-07 / `MINT`). Executing a dual-sign mint modifies token supply
in exactly the same way as a single-sign mint — both produce a
coinbase tx that grows the outstanding total. The existing
convention (`domain/permissions.py:112` and BR-WL-07) keeps
supply-mutating actions out of the ADMIN baseline so that every
admin who can execute a mint has done so via a per-user grant in
`user_permissions`, which writes an `audit_log` row. Putting
`APPROVE_TREASURY_MINT_OP` in the baseline would invert this: a
dual-sign mint would become **easier** than a single-sign mint
(no per-admin grant required for either approver), which defeats
the purpose of the dual-sign envelope. The other "approve" perm
(`APPROVE_TREASURY_DISTRIBUTION`) moves existing tokens between
wallets without touching supply, so it remains in the ADMIN
baseline.

Permission grants follow the standard override path
(`MANAGE_PERMISSIONS`).

## 6. Business rules (BR-TR-*)

Stubs land in `docs/business-rules.md §8e` referencing this spec; full
text lands per-rule with the code in 7.8.x.

| ID | Rule (one-liner) |
|----|------|
| BR-TR-01 | `SAME_SIGNER_FORBIDDEN` — approver.user_id ≠ initiated_by, enforced at service AND DB constraint, **independently per table** (`chk_dist_same_signer` on `treasury_distributions`, `chk_mint_same_signer` on `treasury_mint_ops`). |
| BR-TR-02 | Distribution recipients must each have a non-treasury wallet in the op currency; missing → `RECIPIENT_NO_WALLET`. |
| BR-TR-03 | Distribution duplicates inside `recipient_user_ids` → `RECIPIENTS_DUPLICATE` (no auto-dedup, fail fast). |
| BR-TR-04 | Approval is atomic with execution; partial execution forbidden. Failure on submission rolls back any prior submissions and leaves the op `pending_approval`. Applies to both distributions (N transfers) and mint ops (1 coinbase). |
| BR-TR-05 | `executed` and `cancelled` are terminal — any attempt to mutate returns `DISTRIBUTION_NOT_PENDING` / `MINT_OP_NOT_PENDING` (per entity). |
| BR-TR-06 | Only the initiator can cancel a pending op (`NOT_INITIATOR` otherwise). Same rule for both entities. |
| BR-TR-07 | Mint threshold: `amount >= MINT_DUAL_SIGN_THRESHOLD` (config) routes mint through `treasury_mint_ops`; below threshold the existing synchronous behavior is preserved bit-for-bit (no row in `treasury_mint_ops`). |
| BR-TR-08 | Supply conservation holds at the chain level: a distribution emits exactly N transfers totalling `N * amount_per_wallet`; a dual-sign mint emits exactly one coinbase identical to today's mint. The envelope adds no supply, only gating. |
| BR-TR-09 | Audit emits a per-entity lifecycle: distributions → `TREASURY_DISTRIBUTION_{INITIATED,APPROVED,EXECUTED,CANCELLED}`; mint ops → `TREASURY_MINT_OP_{INITIATED,APPROVED,EXECUTED,CANCELLED}`. Severity (BR-AD-10): the four `EXECUTED` events → `critical`; the rest → `warning`. The existing synchronous `MINT` action stays unchanged. |
| BR-TR-10 | `GET /admin/treasury/distribute` and `GET /admin/mint/operations` each return only their own entity — ADMIN + OPERATOR see all rows; no further row-level scoping in this phase. |

```python
ACTION_TREASURY_DISTRIBUTION_APPROVED:  Final[str] = "TREASURY_DISTRIBUTION_APPROVED"
ACTION_TREASURY_DISTRIBUTION_EXECUTED:  Final[str] = "TREASURY_DISTRIBUTION_EXECUTED"

# Mint-op lifecycle (above-threshold mints only; existing synchronous MINT stays)
ACTION_TREASURY_MINT_OP_INITIATED: Final[str] = "TREASURY_MINT_OP_INITIATED"
ACTION_TREASURY_MINT_OP_APPROVED:  Final[str] = "TREASURY_MINT_OP_APPROVED"

ACTION_TREASURY_MINT_OP_CANCELLED: Final[str] = "TREASURY_MINT_OP_CANCELLED"
```

Payload (`details` JSON) per event:

- `TREASURY_DISTRIBUTION_INITIATED`: `{op_id, currency, amount_per_wallet, recipient_count, source_wallet_id}`
- `TREASURY_DISTRIBUTION_APPROVED`: `{op_id, approved_by}`
- `TREASURY_DISTRIBUTION_EXECUTED`: `{op_id, tx_ids}`
- `TREASURY_DISTRIBUTION_CANCELLED`: `{op_id, by}`
- `TREASURY_MINT_OP_INITIATED`: `{op_id, currency, amount, target_wallet_id, reason}`
- `TREASURY_MINT_OP_APPROVED`: `{op_id, approved_by}`
- `TREASURY_MINT_OP_EXECUTED`: `{op_id, tx_id}`
- `TREASURY_MINT_OP_CANCELLED`: `{op_id, by}`

`target_id` on every entry is the `op_id`.

Severity map (`docs/business-rules.md` BR-AD-10 update):

- `critical`: `TREASURY_DISTRIBUTION_EXECUTED`, `TREASURY_MINT_OP_EXECUTED` (parity with `MINT`)
- `warning`: every `*_INITIATED`, `*_APPROVED`, `*_CANCELLED`

## 10. Test plan

Unit (per service method, in-memory store):

- `initiate_distribution` happy path + each validation error code.
- `approve` happy path (N=1, N=3) + `SAME_SIGNER_FORBIDDEN` +
  `TREASURY_OP_NOT_PENDING` (re-approve after executed; approve after
  cancelled) + insufficient funds at execute time (recheck).
- `approve` atomicity: simulate failure on the 2nd of 3 transfers and
  assert (a) op stays `pending_approval`, (b) the 1st transfer is no
  longer in mempool, (c) `executed_tx_ids` stays `NULL`.
- `cancel` happy path + `NOT_INITIATOR` + double-cancel
  (`TREASURY_OP_NOT_PENDING`).
- Mint dual-sign: below threshold preserves current response (201, same
  fields); above threshold returns 202 + op_id; approve executes the
  coinbase identical to today's mint (same tx_id format, same audit
  `MINT` action chained to `TREASURY_OP_EXECUTED`).

Integration (PG adapter, `-m integration` marker):

- Migration applies cleanly on a fresh DB and is idempotent.
- `chk_same_signer` DB constraint rejects direct DB-level violation
  (defence-in-depth test).
- Round-trip persistence: initiate → approve survives a process restart
  (read back from PG).

Route-level (Quart test client):

- Permission gating: missing `INITIATE_TREASURY_OP` → 403; missing
  `APPROVE_TREASURY_OP` → 403.
- Response shape matches §2 exactly (field-by-field).
- Audit entries land in the order INITIATED → APPROVED → EXECUTED
  with the expected `details` payload.

Coverage target: 80% (project gate). New service modules should
comfortably exceed that given the branchy validation logic.

## 11. Rationale — two tables instead of one

The initial draft modelled both flows as one `treasury_operations`
table with a `kind ∈ {DISTRIBUTE, MINT}` discriminator and a unified
`GET ?kind=` endpoint. That was rejected in review. The argument:

- **Different inputs.** DISTRIBUTE carries `source_wallet_id`,
  `amount_per_wallet`, `recipient_user_ids`. MINT carries
  `target_wallet_id`, `amount`, `reason`. No row has all six columns
  populated — three are always NULL.
- **Different post-conditions.** DISTRIBUTE preserves total supply
  (transfer). MINT increases it (coinbase). Conflating them in one
  table means every query that touches "treasury ops" has to
  remember which kind it cares about and where the per-shape rules
  live.
- **Different validation.** Recipients, treasury source, currency
  match are distribution-only. Mint threshold is mint-only. A unified
  service would carry two independent validation paths gated by a
  `match kind` switch — i.e. two services pretending to be one.
- **Same envelope ≠ same entity.** The dual-sign columns
  (`initiated_*`, `approved_*`, `executed_*`, `cancelled_*`) repeat
  in both tables. That repetition is the **cheapest** way to
  preserve the envelope semantics per entity; an extracted
  `treasury_signers` table would be premature normalisation for 1+1
  signers (revisit when / if N-of-M arrives, §12 follow-up).

If a future phase needs a unified feed (e.g. "all treasury activity
last 24h"), the right answer is a SQL view or a thin read-model that
`UNION ALL`s the two tables — not re-merging them.

## 12. Out-of-scope / follow-ups

- **Reject action.** Approver may want to refuse without forcing the
  initiator to cancel. Tracked for a follow-up phase.
- **N-of-M signers.** Today's contract is strictly 1+1.
- **TTL on pending ops.** Stale `pending_approval` rows accumulate.
- **Email / push notification to approvers.** Out of scope; the
  frontend polls + WebSocket suffices for 7.8 frontend.
- **Per-row read scoping on `GET`.** ADMIN + OPERATOR both see all ops.
  No initiator-only filtering.
- **Per-recipient amount.** Distribution sends the same
  `amount_per_wallet` to every recipient. Variable amounts → future
  phase (would change the body shape to `recipients: [{user_id, amount}]`).
- **`PATCH` to edit a pending op.** Cancel + re-initiate is the only
  edit path in this iteration.
