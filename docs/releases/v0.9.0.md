# Release v0.9.0

## Summary

Adds a read-side endpoint for confirmed transactions, closing the persistence loop that was missing in earlier releases. Mining had always moved transactions from `mempool` to `transactions` for durability, but no API surface exposed that table — clients (notably the Vue dashboard) could only learn about confirmations through the live `block_mined` WebSocket event, so refreshing the browser wiped the displayed history. The new `GET /api/v1/transactions` returns the full ordered history joined with each block's timestamp, restoring traceability across restarts and reloads.

## Included Changes

- **API:** `GET /api/v1/transactions` returns `{ transactions: [{ sender, receiver, amount, block_index, block_timestamp }], count }`, ordered by `block_index` ascending and by insertion order within each block.
- **Domain:** extend `BlockRepositoryProtocol` with `get_confirmed_transactions()` so any future repository — in-memory, Postgres, or otherwise — must surface confirmed history.
- **In-memory repo:** `InMemoryBlockRepository` now records confirmed entries alongside blocks; previously `save_confirmed_transactions` was a no-op, which silently drained tests of useful assertions.
- **Postgres repo:** `PostgresBlockRepository.get_confirmed_transactions()` issues a single SQL `SELECT … FROM transactions JOIN blocks ON blocks.index = transactions.block_index ORDER BY block_index, id`, so the timestamp comes from the block itself and there is no risk of confirmation drift.
- **Service layer:** `BlockchainService.confirmed_transactions()` exposes the repository call to the API layer without leaking infrastructure types.
- **Tests:** unit, integration (PostgreSQL via `clean_db` fixture), and HTTP-level tests covering the empty case, the post-mining case, and the persistence case (mempool empty + history retained).

## Validation

- Unit suite: 96 tests pass under `PYTHONPATH=. py -m pytest -q -m "not integration"`.
- Coverage: 86% total (gate: 80%).
- Integration suite (PG): `test_postgres_repository_returns_confirmed_transactions` validates a fresh repo instance can read history written by a previous instance — covers the "service restart" scenario.
- Manual verification: with an empty chain, `GET /api/v1/transactions` returns `{ transactions: [], count: 0 }`. After two `POST /transactions` plus one `POST /mine_block`, the same endpoint returns both records with matching `block_index` and a non-empty `block_timestamp`.

## Notes

- This release pairs with `basic-blockchain-frontend@v0.5.0`, which calls the new endpoint from `MempoolView.onMounted`. Without that frontend release, the UI Transaction History panel still hydrates only from WebSocket events and behaves as before — no breaking change.
- No data-model migrations. The `transactions` table introduced in V004 already had the right shape; only its read path was missing.
- The new endpoint is ungated — it does not consume the mining rate-limit budget and has no body-size constraints. If the chain grows large enough that the response becomes expensive, future work should add cursor-based pagination (`?after_block_index=N`).
- Phase G debt remains closed; no Phase H scope was touched in this release.
