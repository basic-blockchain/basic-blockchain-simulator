# Release v0.11.0

## Summary

First sub-phase of Phase I — Authentication foundation. Adds the user
identity layer the simulator was missing: registration with a one-shot
activation code, password activation under bcrypt, login that issues
a Bearer JWT (HS256, 30-minute TTL by default), and a global Quart
middleware that decodes the token on every request and exposes
`g.current_user` to the routes that will guard themselves in Phase
I.2 (RBAC) and Phase I.3 (wallets, signed transfers).

This release **does not yet enforce auth on existing endpoints** — `POST
/transactions`, `POST /mine_block` and friends keep their v0.10.0
contracts so the v0.6.0 frontend continues to work unchanged. The next
sub-phases close that gate.

The full Phase I sprint plan, with locked design decisions and
work-item tables for I.1 through I.4, lives in
[`docs/phases/phase-i.md`](../phases/phase-i.md). v0.11.0 corresponds to
the I.1 row of that plan; sub-items I.1.1 through I.1.8 are documented
in commits ending with `[I.1.x]`.

## Included Changes

- **Migrations.** V007 creates `users` (32-hex `user_id` PK, unique
  `username`, optional `email`, timestamps). V008 creates the
  `user_role` enum (`ADMIN`/`OPERATOR`/`VIEWER`), `user_credentials`
  (bcrypt `password_hash`, one-shot `activation_code`, `activated_at`,
  `must_change_password`), and the `user_roles` PK on
  `(user_id, role)`.
- **Domain.** `domain/auth.py` introduces `Role` (+ `DEFAULT_ROLE =
  VIEWER`), `hash_password`/`verify_password` over bcrypt with
  configurable rounds, `generate_activation_code` and
  `generate_user_id`, `create_jwt`/`decode_jwt` (HS256), and a typed
  `AuthError(code=...)` for the HTTP layer to map onto 401 responses.
- **Persistence.** `domain/user_repository.py` defines
  `UserRepositoryProtocol` and an `InMemoryUserStore` for tests;
  `infrastructure/postgres_user_store.py` is the matching PG adapter
  and translates Postgres `UNIQUE` violations on `username`/`email`
  into the `UsernameTakenError`/`EmailTakenError` domain errors.
- **HTTP middleware.** `api/auth_middleware.py` registers a global
  `before_request` hook. Public paths (`/`, `/api/v1/`,
  `/api/v1/health`, `/api/v1/chain`, `/api/v1/valid`, `/api/v1/auth/*`,
  the legacy `/get_chain` / `/valid`) flow through with
  `g.current_user = None`. Other paths decode `Authorization: Bearer
  <jwt>` if present and abort with 401 on a tampered or expired token,
  carrying `AUTH_INVALID_TOKEN` / `AUTH_EXPIRED_TOKEN`. A
  `require_auth()` helper aborts with 401 / `AUTH_REQUIRED` for routes
  that need an authenticated identity.
- **HTTP routes.** `api/auth_routes.py` exposes `POST /auth/register`
  (creates the user, emits an activation code, applies the
  bootstrap-admin rule), `POST /auth/activate` (consumes the code and
  sets the bcrypt hash, requires a password ≥ 8 chars), `POST
  /auth/login` (uniform `AUTH_INVALID_CREDENTIALS` for missing user /
  not-activated / wrong password to avoid account enumeration), and
  `GET /auth/me` (returns the decoded identity).
- **App factory.** `create_app` accepts an optional
  `users: UserRepositoryProtocol` (defaults to PG when DSN is set,
  in-memory otherwise), installs the auth middleware, and nests the
  `/auth` blueprint under `/api/v1`. Refuses to start when
  `JWT_SECRET` is empty unless `TESTING=true`.
- **Configuration.** `config.py` gains `JWT_SECRET`, `JWT_ALGORITHM`,
  `JWT_TTL_SECONDS`, `BCRYPT_ROUNDS`, and `BOOTSTRAP_ADMIN_USERNAME`.
  `requirements.txt` adds `PyJWT>=2.8` and `bcrypt>=4.1`.
- **Tests.** New `tests/test_auth.py` (25 cases) covers the domain
  primitives, the in-memory store invariants, the full HTTP flow, and
  the bootstrap-admin promotion (with a `monkeypatch` of the env var).
  `conftest.py` opts the test process into `TESTING=true` before any
  project module imports `config.py`.
- **Docs.** README updated; new "Authentication" section in
  `docs/api-reference.md` documenting every error code the auth flow
  emits; `docs/data-model.md` carries the new `users`,
  `user_credentials`, and `user_roles` ER blocks plus the dataclass
  shapes; `docs/business-rules.md` adds BR-AU-01..07.

## Validation

- `PYTHONPATH=. python -m pytest -q -m "not integration"` — **132 passed,
  15 deselected**, coverage **81.79%** (gate 80%).
- Manual end-to-end (`TESTING=true`, in-memory store):
  ```
  GET  /api/v1/chain                            -> 200 (public)
  POST /api/v1/auth/register {username,…}       -> 201 + activation_code
  POST /api/v1/auth/activate {code, password}   -> 200
  POST /api/v1/auth/login    {username, passw}  -> 200 + access_token
  GET  /api/v1/auth/me        (no token)        -> 401
  GET  /api/v1/auth/me        (Bearer)          -> 200 + roles=[VIEWER]
  GET  /api/v1/auth/me        (tampered Bearer) -> 401
  POST /api/v1/auth/login    {wrong password}   -> 400 INVALID_CREDENTIALS
  ```

## Migration / breaking notes

- **No data destruction.** V007 / V008 only create new tables and the
  `user_role` enum. Existing `blocks`, `transactions`, `mempool`, and
  `nodes` rows are unaffected.
- **Operator action required.** Production deployments must set
  `JWT_SECRET` (recommended: 32+ random bytes) before upgrading. The
  app refuses to start otherwise. `BOOTSTRAP_ADMIN_USERNAME` is
  optional but recommended so the first registered user becomes ADMIN
  without a manual DB poke.
- **No API breaking change.** All v0.10.0 endpoints (`/chain`,
  `/valid`, `/transactions`, `/mine_block`, `/health`, etc.) keep
  their pre-Phase-I shapes and authentication is not yet enforced on
  them. The frontend (`basic-blockchain-frontend@v0.6.0`) continues
  to work unchanged. Phase I.3 will gate write endpoints behind the
  new identity layer.

## Notes

- The wallet-side cryptography (BIP-39 mnemonic + ECDSA secp256k1
  signature for transfers) is an explicit Phase I.3 deliverable. See
  the locked decision #2 in [`docs/phases/phase-i.md`](../phases/phase-i.md).
- The single-code login response (`AUTH_INVALID_CREDENTIALS`) is
  intentional: distinguishing "no such user" from "wrong password" or
  "not yet activated" would let an attacker probe the username space.
- The middleware deliberately does **not** force every request to carry
  a token. Endpoints that need one call `require_auth()` themselves;
  Phase I.2 will add `@require_permission(...)` on top of the same
  `g.current_user`.
