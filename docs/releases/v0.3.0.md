# Release v0.3.0

## Summary

Adds persistent mempool via `PostgresMempoolRepository`, completing the repository-injection pattern for both blocks and mempool. Pending transactions now survive process restarts. `create_app()` accepts a `dsn` parameter to wire Postgres repositories end-to-end in production, while the in-memory default is preserved for unit tests.

## Included Changes

- Add `MempoolRepositoryProtocol` (typing.Protocol) and `InMemoryMempoolRepository` to the domain layer; refactor `MempoolService` to accept repository injection with in-memory default.
- Implement `PostgresMempoolRepository` backed by the existing `mempool` table with atomic `flush()` (SELECT + DELETE in the same transaction) and FIFO ordering by `(created_at, id)`.
- Extend `create_app()` with an optional `dsn` parameter; when provided, both `PostgresBlockRepository` and `PostgresMempoolRepository` are wired automatically.
- Update `__main__` to pass `dsn=DATABASE_URL` so production runs use full PostgreSQL persistence.
- Add 4 `@pytest.mark.integration` tests: transaction survival across new service instances, flush clearing the DB, FIFO ordering on flush, and count reflecting live DB state.

## Validation

- 17 unit tests pass (`PYTHONPATH=. py -m pytest -q -m "not integration"`).
- Coverage: 82% total (gate: 80%).
- 9 integration tests deselected when `DATABASE_URL` is not set (safe for CI without DB).

## Notes

- `PostgresMempoolRepository` is a prerequisite for Phase G (multi-node mempool sharing).
- In-memory default in `create_app()` ensures zero breaking changes to existing unit tests.
- `flush()` atomicity guarantees no transaction is lost between SELECT and DELETE under concurrent access.
