# Release v0.5.0

## Summary

Adds multi-node consensus (Phase G.2): nodes can register peer URLs, fetch their chains, and converge on the longest valid chain via a REST pull model. Introduces `NodeRegistryProtocol`, `InMemoryNodeRegistry`, and `ConsensusService` in the domain layer, three new API endpoints, and an architecture decision document.

## Included Changes

- Add `NodeRegistryProtocol` (typing.Protocol) and `InMemoryNodeRegistry` — URL normalisation (handles missing `://`), deduplication via set, sorted listing.
- Add `ConsensusService.resolve()` — fetches `/api/v1/chain` from each registered peer, validates with `is_valid_chain()`, and replaces local chain when a longer valid one is found (longest-chain rule).
- Extend `BlockchainService` with `is_valid_chain(blocks)` and `replace_chain(blocks)`; add `replace_all(blocks)` to `BlockRepositoryProtocol` with implementations in `InMemoryBlockRepository` and `PostgresBlockRepository`.
- Add `POST /api/v1/nodes/register` — registers one or more peer URLs; `GET /api/v1/nodes` — lists peers; `GET /api/v1/nodes/resolve` — triggers consensus and returns whether the local chain was replaced.
- `create_app()` accepts optional `node_registry` parameter for test injection.
- Add `docs/architecture/phase-g2-consensus.md` — four design decisions with rationale, rejected alternatives, limitations table, and component diagram.
- 24 new unit tests (13 domain + 11 API); total suite 48 tests.

## Validation

- 48 unit tests pass (`PYTHONPATH=. py -m pytest -q -m "not integration"`).
- Coverage: 81% total (gate: 80%).
- Node registration, listing, and resolve endpoints verified with mocked remote chains.

## Notes

- Peer registry is in-memory only — lost on process restart. PostgreSQL-backed registry is deferred to Phase G.3.
- `resolve()` uses a pull model (GET /chain from each peer). Push notification on new block is deferred to Phase G.3 along with mempool propagation.
- See `docs/architecture/phase-g2-consensus.md` for full design rationale and known limitations.
