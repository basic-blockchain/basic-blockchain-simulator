# Release v0.10.0

## Summary

Phase H+ closes a real integrity gap that v0.9.0 left open. Until now, `Block` was header-only and `hash_block` did not cover transactions — so a row in the `transactions` table could be edited after the fact and `is_chain_valid()` would still return `True`. v0.10.0 makes `Block` carry its transactions as a first-class field, stamps each block with a Merkle root over those transactions, and includes the Merkle root in the chain hash. Tampering with any confirmed transaction now flips chain validity to `False`.

This release pairs with `basic-blockchain-frontend@v0.6.0`, which consumes the new `Block` shape (`merkle_root` + nested `transactions[]`).

The full plan, locked design decisions, API contract, and Track A / Track B work-item tables are documented in [`docs/phases/phase-h-plus.md`](../phases/phase-h-plus.md).

## Included Changes

- **Domain.** `Block` gains `merkle_root: str` and `transactions: list[Transaction]`. New module-level `_compute_merkle_root(txs)` builds a binary sha256 Merkle tree (Bitcoin convention: duplicate the last hash on odd-sized levels). Empty input yields the documented constant `EMPTY_MERKLE_ROOT = sha256("").hexdigest()`. `BlockchainService.create_block(proof, previous_hash, transactions=None)` stamps the block with both fields. `hash_block` switches to an explicit header payload (`{index, timestamp, proof, previous_hash, merkle_root}`) instead of `to_dict()`, so the chain hash covers the Merkle root without redundantly hashing every transaction body.
- **Validation.** `_validate_blocks` now recomputes each block's Merkle root from its actual transactions and rejects the chain on mismatch. Combined with the new hash payload, post-hoc edits to the `transactions` table or to `block.merkle_root` are detectable.
- **Mining flow.** `_mine` flushes the mempool **before** calling `create_block` so the new block can be stamped with the right Merkle root. The mining response now also exposes `merkle_root`. The existing top-level `transactions` field is retained for back-compat with v0.9.0 clients.
- **Persistence.** `migrations/versions/V006__block_merkle_root.sql` truncates the existing chain (acceptable: development simulator with no production data, user-authorised) and adds `merkle_root TEXT NOT NULL` to `blocks`. `PostgresBlockRepository.append` writes the block row and its transactions in the same DB transaction; `get_all` and `last` hydrate `Block.transactions` via a `LEFT JOIN transactions` query. `replace_all` is updated symmetrically so consensus chain replacement does not orphan transactions.
- **Single source of truth.** `BlockchainService.confirmed_transactions()` now derives its result from `chain.blocks[*].transactions` rather than the `_confirmed` side-channel. The `transactions` table is the only place confirmed transactions live; `Block.transactions` is hydrated from it at read time.
- **Tests.** New `tests/test_phase_h_plus.py` covers Merkle determinism, ordering sensitivity, odd-count handling, tamper detection (transaction amount and merkle_root mutation), HTTP `/chain` exposure of new fields, and PG round-trip. Existing tests that used the v0.9.0 split pattern (`create_block` then `save_confirmed_transactions`) updated to pass `transactions=` to `create_block`.
- **Docs.** README badges to v0.10.0; `docs/api-reference.md` documents `merkle_root` plus nested `transactions[]` in mining and chain responses; `docs/data-model.md` updates `Block` and the `blocks` ER diagram; `docs/business-rules.md` adds BR-BL-11, BR-BL-12, and BR-CH-05.

## Validation

- **Unit suite.** `PYTHONPATH=. python -m pytest -q -m "not integration"` — 107 passed, coverage 83.06% (gate 80%).
- **Integration suite.** `PYTHONPATH=. python -m pytest -q -m integration` — runs against PostgreSQL via the `clean_db` fixture. Two new cases (`test_postgres_repository_round_trips_block_with_transactions`, `test_postgres_chain_validates_after_restart`) plus the existing PG suite updated to the new pattern.
- **Manual verification.**
  - Start `python basic-blockchain.py` → `POST /api/v1/transactions {"sender":"alice","receiver":"bob","amount":1.0}` → `POST /api/v1/mine_block`. Response includes `merkle_root` and the new `transactions` is also nested under each block in `GET /api/v1/chain`.
  - Tamper test (in-memory): mutate `service.chain[-1].transactions[0].amount`, call `service.is_chain_valid()` → returns `False`.

## Migration notes

- **Breaking on disk.** V006 truncates the `blocks` and `transactions` tables. Genesis is recreated with the new shape on startup. There is no production data to preserve, but operators with local Postgres instances will lose their development chain on this upgrade.
- **API back-compat.** v0.9.0 clients reading the top-level `transactions` field of `POST /api/v1/mine_block` continue to work. New clients should prefer the nested `block.transactions` (also available on every chain entry).
- **`save_confirmed_transactions`** is kept on the protocol and on both repository implementations for backward compatibility with external callers, but production code (`_mine`) no longer calls it. The block + its transactions land atomically through `repo.append`.

## Notes

- The new Merkle root design uses a simple binary tree without inclusion proofs. SPV-style partial proofs are deliberately out of scope for v0.10.0 and remain a possible follow-up that does not require changing the on-chain hash format.
- The chain hash now distinguishes "empty block" (no txs, `merkle_root = EMPTY_MERKLE_ROOT`) from "no block" — useful for any consumer that wants to trust the empty case explicitly.
