# Release v0.13.0

## Summary

Phase I.3 — wallets, balances, signed transfers. The simulator now
implements the Web3 wallet pattern end-to-end: each wallet owns a
secp256k1 keypair derived from a BIP-39 mnemonic generated at creation,
the server persists only the public key, and every transfer carries a
hex ECDSA signature plus a strictly-monotonic nonce. The chain
validator re-verifies every signature on every walk, so a tampered
`transactions` row makes `is_chain_valid()` return `false`. Total
supply is conserved across mints and transfers — the only way coin
enters the system is `POST /admin/mint`, gated by `MINT` (which is
**not** in the ADMIN baseline; an admin must self-grant it first).

This release closes the I.3 row of the Phase I plan
([`docs/phases/phase-i.md`](../phases/phase-i.md)). Phase I.4 will
adopt the new endpoints from the Vue frontend.

## Included Changes

- **Migrations.** V011 introduces `wallets` (32-hex `wallet_id` PK,
  user FK, `currency` defaulting to `NATIVE`, `balance NUMERIC(28,8)`,
  `public_key TEXT NOT NULL`, `frozen` boolean, timestamps) and
  TRUNCATEs `blocks`/`transactions`/`mempool` so the new on-chain
  shape lands cleanly. `transactions` and `mempool` get four new
  columns (`sender_wallet_id`, `receiver_wallet_id`, `nonce`,
  `signature`). V012 introduces `wallet_nonces` (PK on `wallet_id`,
  `last_used_nonce`, `last_used_at`) for replay protection.
- **Crypto domain.** `domain/crypto.py` (new) wraps `mnemonic` and
  `coincurve`: `generate_mnemonic` (12 words, 128-bit entropy),
  `mnemonic_to_seed` (with checksum validation),
  `derive_keypair` (single keypair per wallet, deterministic),
  `public_key_hex` / `public_key_from_hex`,
  `canonical_transfer_message` (the single source of truth for
  signed bytes), and `sign` / `verify`. `verify` returns False on
  any failure — never raises.
- **Wallet domain.** `domain/wallet_repository.py` defines the
  protocol with typed exceptions (`WalletNotFoundError`,
  `WalletFrozenError`, `NonceReplayError`,
  `InsufficientBalanceError`) plus an `InMemoryWalletStore` for
  tests. `domain/wallet.py` exposes three services —
  `WalletService.create_wallet` returns the mnemonic ONCE and
  `del`s the seed locals after deriving the keypair;
  `TransferService.build_transaction` runs the five-step gate
  (existence, freeze, balance, signature, monotonic nonce) and
  reserves the nonce LAST so a signature-rejected attempt does
  not poison the wallet's sequence; `MintService.build_mint`
  builds a coinbase tx with `signature == "MINT"`.
  Module-level `apply_block_deltas(wallets, transactions)` is
  the single function `_mine` and `_validate_blocks` call to
  apply / re-apply balance changes; legacy txns with empty
  wallet IDs are explicitly skipped for back-compat.
- **Persistence.** `infrastructure/postgres_wallet_store.py`
  (new): `reserve_nonce` is a single SQL UPSERT with a guarded
  WHERE so two concurrent requests on the same nonce serialise
  via the row lock — the loser raises `NonceReplayError`.
  `apply_transfer` selects both wallets `FOR UPDATE` in
  deterministic order to avoid the cross-transfer deadlock
  pattern. `total_supply` powers the conservation invariant
  tests.
- **Transaction model.** `domain/models.py::Transaction` gains
  `sender_wallet_id`, `receiver_wallet_id`, `nonce: int`,
  `signature: str`, all defaulted to empty values so legacy code
  paths still construct. `to_dict` emits all seven fields.
- **HTTP — wallet endpoints.** `api/wallet_routes.py` (new) mounts
  four routes under `/api/v1`: `POST /wallets` (gated by
  `CREATE_WALLET`, returns mnemonic ONCE), `GET /wallets/me`,
  `POST /transactions/signed` (gated by `TRANSFER`,
  ownership-checked, full five-step validation), and
  `POST /admin/mint` (gated by `MINT` — NOT in ADMIN baseline).
  `api/schemas.py` gains `parse_signed_transaction` for the new
  shape; the legacy `parse_transaction` stays for back-compat.
- **Mining.** `_mine` now applies balance deltas via
  `apply_block_deltas` after the block is created. The legacy
  `POST /transactions` path keeps working but its transactions
  no longer move balances — chain history records them but the
  ledger remains in equilibrium.
- **Chain validation.** `_validate_blocks` re-verifies every
  non-coinbase transaction's signature against the sender
  wallet's stored public key on every walk. Combined with the
  Phase H+ Merkle root cover, this means any post-hoc edit to
  `transactions` or `wallets.public_key` flips `is_chain_valid()`
  to false.
- **Tests.** Four new files, 32 new cases:
  * `tests/test_crypto.py` (11) — BIP-39 / secp256k1 / canonical
    message / sign-verify round-trips and tamper detection.
  * `tests/test_wallets.py` (7) — wallet lifecycle endpoints
    plus the explicit-grant-required behaviour for `MINT`.
  * `tests/test_transfers.py` (10) — happy path, every error
    code, replay/decreasing nonce, ownership check, balance
    check.
  * `tests/test_supply_conservation.py` (4) — ledger invariant
    across one transfer, many transfers in one block, many
    transfers across many blocks.
- **Docs.** README badges and surface description; new
  "Wallet endpoints" section in `docs/api-reference.md` with
  the full error-code matrix; `docs/data-model.md` gains the
  `wallets` and `wallet_nonces` ER blocks plus the
  `WalletRecord` dataclass; `docs/business-rules.md` adds the
  new section 7 (BR-WL-01..09); `docs/releases/v0.13.0.md`
  (this file).

## Validation

- `PYTHONPATH=. python -m pytest -q -m "not integration"` — **181
  passed, 15 deselected**, coverage **80.33%** (gate 80%).
- Manual end-to-end (`TESTING=true`, in-memory store):
  ```
  Bootstrap admin → register/activate alice + bob
  POST /wallets (alice, bob)               -> 201 + mnemonic each
  POST /admin/mint without permission      -> 403 FORBIDDEN
  Self-grant MINT to admin
  POST /admin/mint {alice, 100}            -> 201
  POST /mine_block                         -> alice balance 100
  Sign transfer alice→bob 30 nonce 1, locally with alice's mnemonic
  POST /transactions/signed                -> 201
  Replay same nonce                        -> 400 NONCE_REPLAY
  Sign with bob's mnemonic, send as alice  -> 400 SIGNATURE_INVALID
  POST /mine_block                         -> alice 70, bob 30
  GET /valid                               -> {valid: true}
  Tamper transactions.amount in DB         -> GET /valid -> false
  ```

## Migration / breaking notes

- **V011 truncates `blocks`, `transactions`, and `mempool`.** Same
  precedent as V006 in Phase H+: dev-only simulator, no production
  data, user-authorised on 2026-05-07. The chain regenerates a
  fresh genesis on next startup.
- **`POST /api/v1/transactions` keeps the v0.10.0 shape** for the
  v0.6.0 frontend's back-compat path. Those transactions are
  recorded in chain history but do **not** move balances. Phase
  I.4 will switch the frontend to `/transactions/signed` and the
  legacy path becomes deprecated.
- **Operator action — install new deps.** `requirements.txt`
  adds `mnemonic>=0.21` and `coincurve>=20.0`. Run `pip install
  -r requirements.txt` after pulling.
- **Operator action — admin self-grant.** Existing ADMIN users
  cannot mint until they grant themselves the `MINT` permission
  via `POST /api/v1/admin/users/<self>/permissions`. The grant
  is a single API call, audited.

## Notes

- Wallet keypairs are derived **from the first 32 bytes of the
  BIP-39 seed**, single-keypair-per-wallet. A future phase can
  upgrade to BIP-32 derivation paths so one mnemonic backs many
  wallets without changing the on-chain contract — only how the
  keypair is produced.
- The mnemonic is generated server-side in v0.13.0 (the server
  sees the seed during keypair derivation and discards it).
  Phase J / a future phase will move generation to 100%
  client-side (`@scure/bip39` + `@noble/curves`); the
  on-chain contract does not change.
- Freeze/unfreeze of wallets is wired in the wallet repository
  but the dedicated admin endpoint to flip the flag is not
  exposed in v0.13.0. The capability is present (`set_frozen`)
  and the `WALLET_FROZEN` error path is exercised by tests; the
  HTTP shim lands in I.4 alongside the wallet UI.
