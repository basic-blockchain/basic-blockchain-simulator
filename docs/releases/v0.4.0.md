# Release v0.4.0

## Summary

Adds the observability baseline for the blockchain node: a health endpoint with active DB connectivity checking, a metrics endpoint with chain and mempool stats, and request-ID correlation across all structured log events.

## Included Changes

- Add `GET /api/v1/health` — returns `{"status","db","chain_height"}`; HTTP 200 when healthy, 503 when PostgreSQL is unreachable. `db` is `"n/a"` in in-memory mode.
- Add `api/health.py` with `check_db_connectivity()` helper (3-second timeout); add `chain_length()` to `BlockchainService`.
- Add `GET /api/v1/metrics` — returns `chain_height`, `pending_transactions`, and `avg_mine_time_seconds` (mean interval between consecutive blocks; `null` with only the genesis block). Adds `avg_mine_time_seconds()` to `BlockchainService`.
- Add `before_request` hook that assigns `g.request_id` from the `X-Request-ID` header or auto-generates a `uuid4`; `_JSONFormatter` injects `request_id` into every structured log record within a request context.
- 8 new unit tests covering all endpoint states and request-ID propagation.

## Validation

- 25 unit tests pass (`PYTHONPATH=. py -m pytest -q -m "not integration"`).
- Coverage: 81% total (gate: 80%).
- Health and metrics endpoints verified against in-memory mode and mocked DB states.

## Notes

- `GET /api/v1/health` is safe to use as a load-balancer probe: returns 503 on DB failure without exposing internal error details.
- `avg_mine_time_seconds` returns `null` when the chain has only the genesis block — callers should handle this case.
- `request_id` only appears in log records emitted within a Flask request context; background or startup logs are unaffected.
