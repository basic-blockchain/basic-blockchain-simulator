# Release v0.6.1

## Summary

Completes Phase G.5 (async peer dispatch): `PropagationService` now contacts all registered peers in parallel via `ThreadPoolExecutor` instead of serially. Mine and add-transaction latency is now bounded by the slowest single peer rather than the sum of all peer timeouts.

## Included Changes

- Replace synchronous per-peer loop in `broadcast_transaction` and `notify_resolve` with `ThreadPoolExecutor` (up to 8 workers, capped to peer count); all futures are awaited before returning.
- `_post` and `_get` internals unchanged — only the dispatch strategy changes; all existing tests pass without modification.

## Validation

- 62 unit tests pass (`PYTHONPATH=. py -m pytest -q -m "not integration"`).
- Coverage: 80% total (gate: 80%).
- Observable behaviour identical to v0.6.0; parallelism verified through mock call counts.

## Notes

- Worker count capped at 8 and at peer count, whichever is smaller, to avoid over-provisioning threads for small networks.
- G.6 (WebSocket real-time push) remains pending — see `docs/architecture/phase-g3-g4-propagation.md` limitations table.
