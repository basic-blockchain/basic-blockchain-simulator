# Release v0.12.0

## Summary

Phase I.2 — Roles, permissions and audit. Adds the RBAC layer the
Phase I.1 middleware was waiting for: a `Permission` enum with 15
entries, a 3-level resolver (per-user grant > per-role override >
hardcoded baseline), six admin-only HTTP endpoints (list users, grant /
revoke roles, ban / unban, grant / revoke per-user permissions, read
audit), and an append-only audit log that captures every state-mutating
admin action with `actor`, `action`, `target`, and a JSONB details
payload.

The headline architectural decision is **least-privilege by default**.
ADMIN's hardcoded baseline covers user / role / permission management
plus the admin's own wallet ops, but **does not** include
financial-action permissions (`MINT`, `FREEZE_WALLET`,
`UNFREEZE_WALLET`) or cross-user data visibility (`VIEW_WALLETS`,
`VIEW_TRANSFERS`). An ADMIN must explicitly grant those to themselves
through `POST /admin/users/<self>/permissions` — the action is audited.
This prevents a single compromised ADMIN session from instantly
accessing the supply or every user's financial history.

The full Phase I sprint plan (sub-phases I.1 through I.4 plus the Phase
J multi-currency follow-up) lives in
[`docs/phases/phase-i.md`](../phases/phase-i.md). v0.12.0 closes the
I.2 row of that plan.

## Included Changes

- **Migrations.** V009 introduces `permissions` (catalog seeded with 15
  entries), `role_permissions`, `user_permissions`, plus the
  `users.banned BOOLEAN` column. V010 introduces the append-only
  `audit_log` (BIGSERIAL `id`, actor / action / target / JSONB
  details, indexed on actor / target / action / `created_at DESC`).
- **Domain.** `domain/permissions.py` (new) defines `Permission`
  (15 entries spanning user management, audit, and wallet
  operations declared early so the V009 seed is exhaustive),
  `ROLE_PERMISSIONS` baselines (least-privilege ADMIN, audit-light
  OPERATOR, restricted VIEWER), `has_permission(user_id, roles,
  permission, role_overrides, user_overrides)` for the 3-level
  resolution (override REPLACES baseline rather than augmenting),
  and `effective_permissions(role)` for an introspection endpoint.
  `domain/audit.py` (new) defines stable action-name constants and
  the `AuditEntry` read-side dataclass.
- **Persistence.** `domain/user_repository.py` extended with
  `revoke_role`, `list_users`, `set_banned`, `get_role_overrides`,
  `get_user_overrides`, `grant_user_permission`,
  `revoke_user_permission`, `append_audit`, `recent_audit`. Both
  `InMemoryUserStore` and `infrastructure/postgres_user_store.py`
  implement the new surface; the PG store writes audit details
  through `psycopg2.extras.Json` so the JSONB column receives a
  proper Postgres value.
- **HTTP — RBAC plumbing.** `api/permissions.py` (new) exposes
  `@require_permission(Permission.X)` and the
  `set_permission_resolver(role_overrides=, user_overrides=)` seam
  the app factory uses to wire the live persistence loaders. The
  decorator aborts with 401 / `AUTH_REQUIRED` if there is no
  authenticated user and 403 / `FORBIDDEN` on missing permission.
- **HTTP — admin endpoints.** `api/admin_routes.py` (new) mounts six
  routes under `/api/v1/admin`: `GET /users`, `POST
  /users/<id>/roles`, `POST /users/<id>/ban`, `POST
  /users/<id>/unban`, `POST /users/<id>/permissions`, `GET /audit`.
  Each route is gated by the matching `@require_permission(...)`
  and writes an `audit_log` row on success. The ban route refuses
  self-bans with `SELF_ACTION_FORBIDDEN` so the actor cannot lock
  themselves out.
- **HTTP — auth gate updates.** `api/auth_routes.py`'s `login`
  collapses the not-yet-activated and banned cases into the same
  `AUTH_INVALID_CREDENTIALS` response as wrong-password, preserving
  the no-enumeration property (BR-AU-03 + BR-RB-08).
- **App factory.** `basic-blockchain.py` registers the admin
  blueprint, wires `set_permission_resolver(...)` with the user
  store's loader callbacks, and lists the six new endpoints in the
  v1 route index.
- **Tests.** New `tests/test_rbac.py` (17 cases): permission
  resolution semantics including the override-replaces-baseline
  rule and the role-isolation invariant; admin endpoint
  authorisation matrix (no-token / non-admin / admin); admin
  actions including grant/revoke role round-trip, ban/unban
  round-trip with the self-ban guard, per-user permission override
  granting admin-grade access mid-session, and the input-validation
  / unknown-user error paths; banned-account login rejection;
  audit log captures three actions in descending order with the
  right actor / target / details. The pre-existing 132 tests from
  Phase I.1 remain green.
- **Docs.** README badge / surface description / API table
  refreshed; `docs/api-reference.md` adds a full "Admin endpoints"
  section with examples and error codes; `docs/data-model.md`
  carries the four new tables in the ER diagram;
  `docs/business-rules.md` adds BR-RB-01..08 and updates BR-AU-03
  to mention the banned-account collapse.

## Migration / breaking notes

- **No data destruction.** V009 / V010 are additive — they create
  three new tables, one new enum, one new column on `users`, and
  the `audit_log`. Existing rows are unaffected.
- **No public API breaking change.** v0.10.0 / v0.11.0 endpoints
  keep their shapes. The admin endpoints are new.
- **Operational note.** ADMINs upgrading from v0.11.0 keep their
  ADMIN role but their hardcoded baseline narrows: anything they
  used to enjoy through the implicit "ADMIN gets every permission"
  default now requires an explicit grant. The ADMIN can self-grant
  through `POST /admin/users/<self>/permissions` (the
  `MANAGE_PERMISSIONS` permission stays in the baseline so the
  self-elevation works without manual SQL).

## Notes

- The audit log is append-only by convention (no UPDATE / DELETE
  in `domain/audit.py` or the PG adapter). Operators that need a
  retention policy can prune `audit_log` directly with care.
- The ban check runs at login time only. Existing JWTs remain
  valid until they expire — full server-side session revocation on
  ban is intentionally out of scope for v0.12.0; a short JWT TTL
  (default 30 minutes) plus the audit trail is the trade-off.
- Phase I.3 (wallets, signed transfers) builds directly on this
  foundation — its endpoints will declare `@require_permission(
  Permission.TRANSFER)` / `Permission.MINT` / `Permission.FREEZE_WALLET`,
  which makes the override / audit story uniform across the
  surface.
